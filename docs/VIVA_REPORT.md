# BookMyHospital Viva Report

## 1) Project in one line

BookMyHospital is a **multi-app hospital management and emergency booking system** with:

- `bmh_client` for patients and hospitals
- `bmh_admin` for adminsx
- `backend` for API, approvals, bookings, complaints, and live updates

The live backend is deployed on Render Free at:

`https://bookmyhospital-api.onrender.com`

---

## 5) Project architecture

### High-level flow

1. Patient or hospital opens the Flutter app.
2. App sends requests to the backend API.
3. Backend stores or reads data from memory or MongoDB.
4. Backend emits Socket.IO events for live refresh.
5. Admin app receives the pending hospital list and can approve or decline.
6. Approved hospitals can log in and update availability.
7. Patients see live hospital data and can book services or submit complaints.

### Main components

- **Client app**: patient + hospital entry points
- **Admin app**: approval and moderation panel
- **Backend**: APIs, persistence, sockets, uploads
- **Cloud hosting**: Render Free backend deployment

## 10) Important runtime behaviors

### Client app behavior

- Stores API URL in `SharedPreferences`
- Uses `API_BASE_URL` from `dart-define` if available
- Falls back to `http://10.0.2.2:8080` for Android emulator
- Uses polling plus sockets for live sync

### Admin app behavior

- Same backend URL storage pattern
- Login is demo-code based for now
- Pending hospitals show in the dashboard
- Approve/decline updates backend instantly

### Backend behavior

- Uses `.env` from repo root
- Reads MongoDB URI when available
- Falls back to memory mode if MongoDB is not configured
- Uses Cloudinary when keys are present
- Falls back to local upload storage otherwise

## 13) Viva-ready answers to likely questions

### Q: What languages did you use?

A: Dart for both Flutter apps, JavaScript for the Node backend, plus YAML and JSON for configs and data exchange.

### Q: Why did you use Flutter?

A: To build cross-platform Android apps with one codebase and fast UI development.

### Q: Why Node.js?

A: It is lightweight, fast for APIs, and works well with Express and Socket.IO for realtime communication.

### Q: Why Socket.IO?

A: To push live updates for bookings, approvals, complaints, and hospital availability.

### Q: What is the backend doing?

A: It handles registration, approval, login, bookings, complaints, statistics, and realtime broadcast events.

### Q: What was installed on Linux?

A: Flutter toolchain, Node.js/npm, Git, GitHub CLI during deployment work, plus Android/Java build support for APK generation.

### Q: What cloud did you use?

A: Render Free for backend hosting, with GitHub auto-deploy.

### Q: What database did you use?

A: MongoDB when configured; otherwise the backend falls back to in-memory demo storage.

### Q: How do complaints store proofs?

A: Locally in the backend uploads folder if Cloudinary is not set, or in Cloudinary when credentials are configured.

### Q: How does the admin approval work?

A: Admin sends a PATCH request to the hospital status endpoint; backend updates the hospital from pending to approved and emits a realtime event.

## 14) What changed recently in the cloud fix

The approval bug was caused by a request mismatch:

- Admin app sent `{ action: 'approve' }`
- Backend initially expected `{ status: 'approved' }`

That was fixed so the service now accepts both formats.
